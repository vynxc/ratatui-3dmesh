# Requirements

Evaluate and refactor the current `ratatui-3dmesh` implementation for human maintainability without changing behavior.

Goals:
- Preserve existing functionality and public UX:
  - OBJ/STL defaults remain available.
  - Optional glTF/GLB support remains behind `gltf`.
  - Optional PNG/JPEG texture support remains behind `textures`.
  - `examples/viewer.rs` keeps current controls, brightness behavior, `--texture`, and terminal lifecycle ownership.
  - Existing commands for `models/model.obj` and `models/axe/scene.gltf` continue to work.
- Improve intuitive structure and DRY boundaries:
  - Make data/model types, loader parsing, render preparation, rasterization, and example UI responsibilities easier to understand.
  - Reduce long functions and repeated color/texture/math conversion code.
  - Avoid introducing broad abstractions, heavyweight dependencies, or framework-like bloat.
- Move toward Rust-idiomatic implementation:
  - Prefer small typed helper structs/functions over tuple-heavy call chains where they clarify intent.
  - Use clear ownership/borrowing boundaries, avoid avoidable clones in hot paths, and keep optional dependencies feature-gated.
  - Keep errors, docs, and tests aligned with public APIs.
- Add a practical clippy pedantic hardening pass:
  - Use `cargo clippy --all-targets --all-features -- -W clippy::pedantic` as a diagnostic input.
  - Fix low-risk pedantic warnings where doing so improves clarity.
  - Add narrow `#[allow(...)]` only for deliberate numeric rendering casts or cases where pedantic advice would make code less readable.

Non-goals:
- Do not add new model format support, rendering features, controls, or dependencies.
- Do not redesign the renderer around a GPU, async pipeline, ECS, plugin framework, or external parser crate.
- Do not change screenshots/output intentionally; any visual differences should be limited to unavoidable bug fixes discovered during refactoring.
- Do not make `clippy::pedantic` a hard CI gate until noisy rendering/math exceptions are explicitly documented and stable.

# Investigation Findings

Current project structure:
- `Cargo.toml` exposes conservative defaults: `default = ["obj", "stl", "mtl"]`; optional `gltf`, `textures`, `serde`, and `cli-example` features.
- `src/lib.rs` has a compact public surface that re-exports config, loader options, model types, widget/state, and optional controls.
- `src/model.rs` is the largest file at ~518 lines and currently combines vector math (`Vec2`, `Vec3`), texture runtime data, materials, faces, mesh construction/loading helpers, bounds, normalization, and tests.
- `src/render/pipeline.rs` is ~441 lines and combines frame preparation, projection, face traversal, texture selection, textured triangle filling, bilinear sampling, color brightening, and tests.
- `src/config.rs` is ~340 lines with public config enums, `Mesh3dConfig`, builder methods, glyph mapping, and tests.
- `src/loader/obj.rs` is ~347 lines and contains parsing, face token/index logic, material loading, and texture attachment.
- `src/loader/gltf.rs` is ~260 lines and contains document import, material extraction, node/primitive traversal, transforms, texture conversion, and tests.
- `examples/viewer.rs` is ~176 lines and remains intentionally dependency-light, but mixes argument parsing, terminal setup/cleanup, app loop, status text, and texture status formatting.

Verified current behavior and recent features:
- `MeshLoadOptions` in `src/loader/mod.rs` has `texture_override`, `load_material_textures`, and `strict_textures` builder methods.
- `Texture` in `src/model.rs` stores decoded RGBA pixels in `Arc<[u8]>` and now caches `average_color` to avoid the previous per-face image scan freeze.
- `ColorMode::{Off, Material, Lighting, Texture, Auto}`, `TextureFilter`, `TextureWrap`, `texture_lighting`, and `color_brightness` are implemented in `src/config.rs` and used by `src/render/pipeline.rs`.
- `src/render/raster.rs` already provides the right low-level shape: simple `fill_triangle` plus closure-based `fill_triangle_with` for textured per-cell painting.
- glTF support in `src/loader/gltf.rs` loads primitives, indices, node transforms, normals, UVs, base-color factors, and base-color textures when `textures` is enabled.
- Docs and wiki files already mention glTF/textures/brightness controls, though minor formatting cleanup is possible.

Clippy pedantic diagnostic snapshot:
- `cargo clippy --all-targets --all-features -- -W clippy::pedantic` currently completes with warnings, not errors.
- High-signal warning categories include:
  - 19 public `Result` functions missing `# Errors` docs.
  - `src/loader/gltf.rs::load_gltf` too many lines (~124/100).
  - `src/render/pipeline.rs` and `src/render/raster.rs` many numeric cast warnings around terminal coordinates, image coordinates, RGB conversion, and glyph indices.
  - `src/controls.rs` has simple `unnested_or_patterns` suggestions for key matches.
  - Several `map(...).unwrap_or_else(...)` and redundant closure suggestions in loader code.
  - Tests have float equality warnings that should become epsilon/range assertions.
  - `MeshLoadOptions` triggers `struct_excessive_bools`; this is acceptable or can be documented because it is a small public options bag.

Primary maintainability risks:
- Hot-path behavior is correct now, but `render_mesh` still clones/normalizes the full `Mesh` every frame when `config.normalize` is true, including texture metadata. This should be reviewed carefully without changing output.
- Loader modules have useful but lengthy functions. Splitting by intent would help humans reason about parsing vs attachment vs conversion.
- Color handling is spread across material conversion, texture sampling, brightness helpers, and lighting mode logic. A small internal color utility would reduce duplication without expanding the public API.
- Numeric casts are inevitable in terminal rasterization, but they need named helper functions or scoped allows so pedantic output distinguishes intentional rendering math from accidental truncation.
- Public docs should explain fallible APIs with `# Errors` sections to satisfy pedantic and improve crate quality.

# Technical Design

Refactor principles:
- Keep public API compatibility unless a change is purely additive and clearly useful.
- Prefer private modules/helpers over new public types unless the type already represents an API concept.
- Keep files discoverable by domain:
  - model/data definitions in `src/model.rs` or small submodules if needed.
  - loader-specific parsing/conversion in `src/loader/*`.
  - render preparation/color/sampling/rasterization in `src/render/*`.
  - example app concerns in `examples/viewer.rs` only.
- Avoid “abstraction for abstraction’s sake”: extract only code that has a name a human would expect, removes duplication, or shortens a function with a clear boundary.

Proposed structure changes:
- Split render pipeline responsibilities internally:
  - Keep `render_mesh(...)` as the public render entry point in `src/render/pipeline.rs`.
  - Introduce small private helpers/structs such as `RenderFrame`, `FaceRenderContext`, or `TrianglePaint` only if they reduce repeated tuple passing.
  - Move color utilities (`brighten_channel`, `brighten_rgb`, `brighten_color`, texture average fallback selection) into a private `src/render/color.rs` or a private section with focused tests.
  - Move bilinear texture sampling or UV sampling helpers into a small private `src/render/texture.rs` module if it meaningfully reduces `pipeline.rs` size.
- Tidy raster numeric conversions in `src/render/raster.rs`:
  - Add private helpers like `round_to_i32`, `point_to_cell`, or `zbuf_index` to centralize bounds checks before `as` casts.
  - Use scoped `#[allow(clippy::cast_possible_truncation, clippy::cast_sign_loss, clippy::cast_precision_loss)]` only where terminal/image coordinate conversion is deliberately lossy after bounds checks.
- Tidy model/data definitions in `src/model.rs`:
  - Group `Vec2`/`Vec3`, texture types, material/face types, `Bounds`, and `Mesh` impl sections with clearer section comments.
  - Keep `Texture::average_color()` cached and ensure tests cover it so future changes do not reintroduce per-frame full-image scans.
  - Consider extracting only vector math into a private `src/math.rs` or `src/model/geometry.rs` if it improves navigation without disrupting public re-exports.
- Tidy loaders:
  - In `src/loader/gltf.rs`, split `load_gltf` into material loading, primitive extraction, face creation, and texture attachment/conversion helpers.
  - In `src/loader/obj.rs`, keep the custom parser but separate line dispatch, face token parsing, material loading, and texture attachment into smaller functions with tests near parser behavior.
  - Apply simple pedantic suggestions (`map_or_else`, redundant closure removal, nested or-patterns) where readability improves.
- Tidy example UX code without changing behavior:
  - Keep manual argument parsing rather than adding `clap`.
  - Extract status text construction and initial config/color selection into named helpers in `examples/viewer.rs`.
  - Preserve raw-mode cleanup behavior and current controls.

Clippy pedantic approach:
- First run pedantic and classify warnings into three buckets:
  1. Fix directly: `unnested_or_patterns`, `map_unwrap_or`, redundant closures, test float comparisons, missing public `# Errors` docs.
  2. Refactor: `too_many_lines` in loader/render functions by extracting named helpers.
  3. Allow narrowly: unavoidable rendering/math casts and intentionally small public bool option structs.
- Prefer local allows on the smallest helper/function rather than crate-level `#![allow(...)]`.
- Add comments on allows explaining why the warning is acceptable, especially for raster coordinate conversion and color channel conversion.
- Keep the existing strict command `cargo clippy --all-targets --all-features -- -D warnings` green throughout. The pedantic command should become substantially cleaner but does not need to be `-D` unless the remaining allows make that realistic.

Potential internal module layout after refactor:

```text
src/
  config.rs              # public config and builder API
  controls.rs            # crossterm key mapping behind cli-example
  model.rs               # public mesh/data types, possibly sectioned or lightly split
  loader/
    mod.rs               # MeshLoadOptions and extension dispatch
    obj.rs               # OBJ line/face parsing + texture attachment
    mtl.rs               # MTL parsing
    stl.rs               # STL parsing
    gltf.rs              # glTF orchestration with smaller helpers
    texture.rs           # image decode behind textures
  render/
    mod.rs
    camera.rs
    raster.rs            # z-buffered primitives and bounds-checked cell math
    color.rs             # private color/brightness helpers, if extracted
    texture.rs           # private sampling helpers, if extracted
    pipeline.rs          # render orchestration
  widget.rs              # Ratatui widget/state/help overlay
examples/
  viewer.rs              # dependency-light demo app
```

Risk controls:
- Avoid broad renames of public items (`Mesh`, `Face`, `Texture`, `Mesh3dConfig`, `MeshLoadOptions`, etc.).
- Avoid changing face triangulation, projection math, UV selection, texture V-flip defaults, or brightness defaults unless a test proves a bug.
- Keep refactors in small, testable batches so any visual/loader regression is easy to isolate.
- Watch performance after touching `render_mesh`; specifically avoid reintroducing per-face texture scans or additional full-mesh clones.

# Delivery Plan
1. Classify pedantic warnings and apply low-risk idiomatic fixes
The codebase has a documented pedantic baseline and simple warnings are removed without behavior changes.
- Run `cargo clippy --all-targets --all-features -- -W clippy::pedantic` and record remaining warning categories by module.
- Update `src/controls.rs`, `src/loader/gltf.rs`, `src/loader/obj.rs`, tests, and small helper call sites for direct readability fixes such as nested or-patterns, `map_or_else`, redundant closure removal, and epsilon/range float assertions.
- Add missing `# Errors` doc sections for public fallible functions in `src/model.rs`, `src/loader/*`, and any public APIs surfaced by `src/lib.rs`.
- Validate that standard non-pedantic clippy with `-D warnings` remains green after each batch.

2. Refactor render internals around named color/sampling responsibilities
Rendering remains visually/functionally equivalent while `src/render/pipeline.rs` becomes easier to read and harder to accidentally slow down.
- Extract repeated brightness/color selection logic from `src/render/pipeline.rs` into focused private helpers or `src/render/color.rs`.
- Keep `Texture::average_color()` as the only texture-average source for wireframe/points fallback and add a regression test for it.
- Centralize UV interpolation and bilinear/nearest sampling helpers so `fill_textured_triangle` contains the flow, not all implementation details.
- Add narrow documented clippy allows for unavoidable color/channel and coordinate casts only inside the helper functions that perform checked conversion.

3. Refactor loader internals into smaller parsing/conversion steps
OBJ and glTF loading keep the same supported formats and feature gates while long functions become navigable by humans.
- Split `src/loader/gltf.rs::load_gltf` into material extraction, primitive extraction, face assembly, transform helpers, and optional texture loading helpers.
- Split `src/loader/obj.rs` line parsing, face token parsing, material loading, and texture attachment boundaries without changing accepted syntax or error behavior.
- Preserve current `MeshLoadOptions` semantics, including `strict_textures`, manual texture override, and MTL texture loading.
- Run all loader tests plus the axe fixture test to confirm supported OBJ/MTL/STL/glTF workflows remain intact.

4. Improve model/config/example organization and documentation polish
Public APIs remain stable while the code is easier for a maintainer to scan and use.
- Reorganize `src/model.rs` and `src/config.rs` with clear sections and helper methods, extracting only small private modules if navigation improves without API churn.
- Tidy `examples/viewer.rs` by extracting initial color/config selection and status text formatting into named helpers while preserving current controls, brightness defaults, and terminal cleanup.
- Fix README/wiki table/list formatting around brightness controls and update docs for any internal organization notes if needed.
- Validate docs with `RUSTDOCFLAGS='-D warnings' cargo doc --all-features --no-deps`.

5. Final regression and pedantic cleanup pass
The refactor is verified against existing functionality and leaves clear guidance for any intentionally remaining pedantic warnings.
