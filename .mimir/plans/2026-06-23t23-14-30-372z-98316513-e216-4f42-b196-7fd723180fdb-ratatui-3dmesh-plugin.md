# Requirements

Build this empty repository into a public Rust crate for embedding an ASCII/terminal 3D mesh viewer inside Ratatui apps.

Goals:
- Convert the core idea of `autopawn/3d-ascii-viewer` into idiomatic Rust: terminal 3D model viewing, OBJ/STL support, ASCII luminance, and optional OBJ/MTL diffuse-color rendering.
- Learn from `luisbedoia/sx3d`: simple Rust CLI usage, terminal-size-aware viewing, and examples for OBJ/STL files.
- Expose a reusable Ratatui “plugin” as a widget crate rather than a standalone-only app.
- Provide many typed configuration options while keeping defaults excellent.
- Include a complete example viewer application and polished public docs, including README and GitHub Wiki source.

Non-goals for the first implementation:
- Texture mapping, skeletal animation, GPU rendering, or arbitrary scene graphs.
- Owning the host app’s terminal lifecycle from the library widget. Terminal initialization belongs in the example app or the embedding application.
- Perfect CAD-grade mesh correctness; target practical viewing of common OBJ/STL files with clear errors.

# Technical Design

Current state: the workspace `/home/vynxc/code/ratatui-3dmesh` is empty, with no `Cargo.toml`, source files, README, or examples. The plan therefore starts from a new crate.

Ratatui integration pattern discovered:
- Ratatui exposes reusable render components through `Widget` and `StatefulWidget` with `render(self, area: Rect, buf: &mut Buffer)` or `render(self, area: Rect, buf: &mut Buffer, state: &mut State)`.
- A 3D viewer needs persistent camera/control state, so the public API should center on `StatefulWidget`.
- For reusable third-party widget crates, the user-facing API should not require taking over the terminal; it should render into a caller-provided `Frame` area.

Proposed crate layout:
```text
Cargo.toml
src/
  lib.rs              # public exports and crate-level docs
  config.rs           # Mesh3dConfig builder and presets
  controls.rs         # optional input-to-state helpers
  error.rs            # crate Result/Error
  model.rs            # Mesh, Vertex, Face, Material, Bounds
  widget.rs           # Mesh3dWidget + Mesh3dState + StatefulWidget impl
  loader/
    mod.rs
    obj.rs
    stl.rs
    mtl.rs
  render/
    mod.rs
    camera.rs
    pipeline.rs
    raster.rs
examples/
  viewer.rs           # complete Ratatui/crossterm demo app
  assets/             # tiny fixtures or properly licensed sample models
docs/wiki/
  Home.md
  Getting-Started.md
  Configuration.md
  Embedding-in-Ratatui.md
  Model-Formats.md
  Roadmap.md
```

Core public API sketch:
```rust
use ratatui_3dmesh::{Mesh, Mesh3dConfig, Mesh3dState, Mesh3dWidget};

let mesh = Mesh::load("model.obj")?;
let config = Mesh3dConfig::default().with_auto_spin(true);
let widget = Mesh3dWidget::new(&mesh).config(config);
frame.render_stateful_widget(widget, area, &mut state);
```

Rendering approach:
- Load OBJ/STL into a normalized internal `Mesh` with vertices, faces, normals, bounds, and optional material color metadata.
- Compute camera transform, projection, lighting intensity, and z-buffer per terminal cell.
- Map intensity to a configurable glyph ramp and optionally apply Ratatui `Style` colors from material diffuse colors or lighting presets.
- Support rendering modes such as solid shaded ASCII, wireframe, points, normals/debug overlays, and auto-fit.

User experience:
- Sensible defaults: auto-fit model, perspective camera, shaded glyph ramp, resize-aware render area, and simple controls.
- Configurable controls/helper API for host apps, plus example bindings: arrows/WASD rotate, `+/-` zoom, `r` reset, `m` render mode, `c` color, `space` pause/auto-spin, `?` help, `q` quit in the example.
- Example app includes status/help panels and robust terminal restore on exit or error.

Risks and mitigations:
- Terminal aspect ratio can distort models: include configurable cell aspect correction and auto-fit.
- Large meshes can be slow: add quality presets, face limits or decimation hooks, culling, and benchmarking-friendly deterministic render tests.
- Material color support varies by terminal palette: use Ratatui true-color styles when available and fall back gracefully.

# Delivery Plan
1. Establish crate API
A new public Rust library crate exists for reusable Ratatui 3D mesh rendering.
- Create `Cargo.toml`, `src/lib.rs`, and module skeletons for `config`, `model`, `loader`, `render`, `widget`, `controls`, and `error`.
- Use `ratatui`/`ratatui-core` widget traits as the public integration surface: `Mesh3dWidget` plus persistent `Mesh3dState` rendered via `StatefulWidget`.
- Add feature gates for optional capabilities such as `obj`, `stl`, `mtl`, `serde`, and `cli-example`, keeping the core widget usable as a library.
- Validate with `cargo check --all-features` once the crate skeleton and public exports are in place.

2. Implement mesh loading
The crate can load and normalize OBJ/STL meshes into an internal renderable model.
- Implement `src/model.rs` data types for vertices, faces, normals, materials, bounds, and loaded `Mesh` metadata.
- Implement `src/loader/obj.rs` and `src/loader/stl.rs` using Rust-native parsers or small focused parsers, covering the reference behavior from `3d-ascii-viewer` and `sx3d`: Wavefront `.obj` and `.stl` inputs.
- Implement optional `src/loader/mtl.rs` diffuse-color parsing for OBJ material libraries, matching autopawn’s color support while documenting texture support as out of scope.
- Add loader tests with tiny inline OBJ/STL/MTL fixtures for face counts, bounds, material assignment, and error messages.

3. Build render pipeline
The renderer converts meshes into terminal cells with configurable projection, lighting, depth, and ASCII/color output.
- Implement `src/render/camera.rs`, `src/render/pipeline.rs`, and `src/render/raster.rs` for camera transforms, perspective/orthographic projection, z-buffering, face shading, and terminal aspect-ratio correction.
- Provide “hella config options” through typed builders in `src/config.rs`: glyph ramp, wireframe/solid/points modes, scale, rotation, camera, light direction, auto-fit, backface culling, color policy, background style, controls hints, and quality/performance presets.
- Use Ratatui `Buffer`, `Rect`, `Style`, and `Color` to render directly into widget areas without owning terminal initialization.
- Validate with deterministic snapshot-style unit tests over a small mesh and fixed viewport, plus numeric tests for projection and bounds normalization.

4. Deliver Ratatui integration and example
The Ratatui user experience exposes ergonomic state, controls, and embeddable examples.
- Implement `src/widget.rs` so app authors can call `frame.render_stateful_widget(Mesh3dWidget::new(&mesh).config(config), area, &mut state)`.
- Implement `src/controls.rs` helpers that map crossterm key events into state changes for rotate, pan, zoom, reset, toggle render mode, toggle color, pause/auto-spin, and help overlay behavior.
- Add `examples/viewer.rs` as a complete Ratatui app with terminal init/restore, CLI path argument parsing, resize handling, error display, status/help panel, and sample controls inspired by the referenced viewers.
- Validate by running `cargo run --example viewer --features cli-example -- <sample-file>` where fixtures or public-domain sample assets are present, plus `cargo test --all-features`.

5. Prepare public documentation
Public project documentation is ready for GitHub and crates.io users.
- Write `README.md` with badges, install/add-to-Cargo instructions, minimal widget snippet, full example command, screenshots/GIF placeholders, supported formats, configuration table, control map, feature flags, limitations, and credits/licenses for referenced inspiration and included sample assets.

- Add `LICENSE`, `CONTRIBUTING.md`, and crate metadata (`description`, `repository`, `keywords`, `categories`, `readme`) suitable for a public release.
