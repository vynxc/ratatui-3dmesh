# Requirements

Add general-purpose texture-image color support to the existing `ratatui-3dmesh` crate without hard-coding the current `models/` assets.

Goals:
- Support common Wavefront texture workflows:
  - OBJ + MTL with `map_Kd` diffuse texture maps.
  - OBJ + UV coordinates + manually supplied texture image when MTL is missing.
  - OBJ + multiple materials/textures, falling back per face when texture data is incomplete.
  - OBJ/MTL diffuse `Kd` colors and grayscale/lighting modes as existing fallbacks.
  - STL and OBJ files without UVs should continue to render with current material/lighting behavior.
- Explicitly support the current asset combination discovered in `models/`:
  - `models/model.obj` has positions, normals, UVs, and faces in `v/vt/vn` form.
  - `models/AXEE_LP_exported_Base_color.jpg` is actually PNG/RGBA image data despite its `.jpg` extension.
  - Missing referenced `model_0.mtl` ... `model_13.mtl` files and zero `usemtl` statements must not prevent using a manual texture override.
- Keep the library idiomatic Rust: typed load options/config, feature-gated optional dependencies, clear errors, no global mutable state, no terminal ownership in library code.
- Keep public UX strong for both library users and the example viewer: easy CLI flags, helpful errors, documented fallbacks, and truecolor terminal output where available.
- Update README and GitHub Wiki source to document texture workflows, limitations, and example commands for public release.

Non-goals for this texture pass:
- GPU rendering, skeletal animation, physically based rendering, normal maps, or full scene-graph support.
- Perfect support for every MTL option. Prioritize `map_Kd`, `Kd`, relative paths, quoted/spaced paths where practical, and graceful ignoring of unsupported material statements.
- Procedurally generating high-quality UV unwraps for models that lack UVs. Provide a clear fallback/error path and optionally a simple future roadmap entry.

# Investigation Findings

Current crate state:
- `Cargo.toml` defines `ratatui-3dmesh` with features `obj`, `stl`, `mtl`, `serde`, and `cli-example`; `image` is not yet present.
- `src/model.rs` currently stores `Mesh { vertices, faces, materials, ... }`, `Face { indices, material, normal }`, `Material { name, diffuse }`, and a small `Vec3` type.
- `src/loader/obj.rs` parses `v`, `f`, `usemtl`, and `mtllib`; it currently ignores `vt` and `vn` face components except for the first vertex index.
- `src/loader/mtl.rs` parses `newmtl` and `Kd`; it does not yet parse `map_Kd`.
- `src/render/pipeline.rs` projects vertices, triangulates faces, computes lighting, chooses glyphs, and calls raster helpers.
- `src/render/raster.rs` already uses barycentric triangle filling and z-buffering, which is the right insertion point for interpolated UV sampling.
- `examples/viewer.rs` accepts one mesh path argument and has keyboard controls/status UI behind the `cli-example` feature.

Current `models/` assets:
- `models/model.obj`: 16,048 vertices, 16,048 UVs, 16,048 normals, 37,296 faces.
- Face syntax is `v/vt/vn`, e.g. `f 1/1/1 2/2/2 3/3/3`, so the OBJ has the UV mapping needed for real texture sampling.
- The OBJ references missing MTL files but has `0` `usemtl` assignments, so manual texture override is the correct compatibility path.
- `models/AXEE_LP_exported_Base_color.jpg` is detected as `PNG image data, 4096 x 4096, 8-bit/color RGBA`; decoding must sniff file contents rather than trusting the extension.

Web/research notes:
- Wavefront OBJ uses `vt` texture coordinates and faces may reference `v/vt/vn`; MTL files commonly connect diffuse texture images through `map_Kd`.
- Common Rust OBJ loaders include `tobj`, `obj-rs`, and `wavefront_obj`; the existing custom parser is small enough to extend, but `tobj` remains a fallback option if parser complexity grows.
- The Rust `image` ecosystem can decode PNG/JPEG into RGBA8. To support mismatched extensions like the current `.jpg` containing PNG bytes, the implementation should read bytes and use format guessing from content before decoding, then normalize to RGBA8.
- Ratatui supports truecolor via `ratatui::style::Color::Rgb(r, g, b)`, which is suitable for per-cell texture color.

# Technical Design

Feature/dependency strategy:
- Add an optional `textures` feature in `Cargo.toml`, for example:
  ```toml
  textures = ["dep:image"]
  image = { version = "0.25", default-features = false, features = ["png", "jpeg"], optional = true }
  ```
- Keep default features conservative. Either include `textures` in defaults only if compile-time impact is acceptable, or document `--features textures` clearly for image color support.
- `cli-example` should not automatically imply textures unless the example needs texture flags compiled in; prefer `cargo run --release --example viewer --features "cli-example textures" -- ...`.

Model/data model changes:
- Add a small `Vec2 { u: f32, v: f32 }` type in `src/model.rs` for OBJ `vt` coordinates.
- Extend `Mesh` with texture-related metadata without breaking non-textured rendering:
  - `tex_coords: Vec<Vec2>`.
  - `textures: Vec<Texture>` behind `textures` feature or a lightweight texture reference structure when the feature is disabled.
  - Optional default/manual texture index for OBJ files with UVs but no material assignments.
- Extend `Face` so each face corner can preserve UV indices. Minimal compatible shape:
  ```rust
  pub struct Face {
      pub indices: Vec<usize>,
      pub tex_coord_indices: Vec<Option<usize>>,
      pub normal_indices: Vec<Option<usize>>,
      pub material: Option<String>,
      pub normal: Option<Vec3>,
  }
  ```
  `tex_coord_indices.len()` should match `indices.len()` for parsed OBJ faces; STL and untextured OBJ can use `None` entries.
- Extend `Material` with optional diffuse texture metadata:
  ```rust
  pub diffuse_texture: Option<TextureRef>,
  ```
  where `TextureRef` can hold the resolved path and loaded texture index/name depending on feature gating.

Texture loading API:
- Add `MeshLoadOptions` in `src/loader/mod.rs` or `src/model.rs`:
  ```rust
  pub struct MeshLoadOptions {
      pub texture_override: Option<PathBuf>,
      pub load_material_textures: bool,
      pub strict_textures: bool,
      pub flip_texture_v: bool,
  }
  ```
- Add ergonomic constructors:
  - `Mesh::load_with_options(path, MeshLoadOptions::default())`.
  - `Mesh::load_textured(path, texture_path)` behind `textures` for the no-MTL/manual override case.
  - Existing `Mesh::load(path)` remains and preserves current behavior.
- Implement texture decode in a new `src/loader/texture.rs` behind `textures`:
  - Read bytes from disk.
  - Guess image format from bytes/content instead of extension.
  - Convert decoded image to RGBA8.
  - Store `width`, `height`, `path`, and packed RGBA pixels.
  - Return clear `Error::TextureLoad` / `Error::TextureDecode` variants with paths.
- Resolve paths relative to the file that names them:
  - `map_Kd` paths are relative to the `.mtl` file directory.
  - Manual override paths are relative to the current process unless already absolute, matching CLI convention.

OBJ/MTL parser changes:
- In `src/loader/obj.rs`, parse:
  - `vt u v [w]` into `Mesh::tex_coords`.
  - `vn x y z` into normal storage or at least preserve normal indices for future use.
  - face tokens in all common forms: `v`, `v/vt`, `v//vn`, `v/vt/vn`, including negative indices.
- Keep triangulation behavior as render-time fan triangulation, but preserve UV indices per polygon corner so triangle UVs are selected with the same corner order.
- In `src/loader/mtl.rs`, parse `map_Kd` and preserve robust path strings:
  - Handle simple paths like `map_Kd texture.jpg`.
  - Ignore common option flags such as `-o`, `-s`, `-bm`, `-clamp` enough to find the texture filename.
  - Support paths with spaces when practical by joining remaining non-option tokens.
- Missing MTL files should remain non-fatal when the current behavior already ignores `Io` for MTL loading, but missing explicitly referenced texture files should produce a warning-like fallback unless `strict_textures` is true.

Rendering changes:
- Extend `ColorMode` in `src/config.rs` with texture-aware variants, for example:
  - `Texture` — use sampled texture color when available, otherwise fallback style.
  - `TextureMaterial` or `Auto` — prefer texture, then material diffuse, then lighting/foreground.
- Add texture sampling config:
  - `texture_filter: TextureFilter` with `Nearest` and `Bilinear`.
  - `texture_wrap: TextureWrap` with `Repeat` and `Clamp`.
  - `texture_lighting: bool` to multiply sampled RGB by lighting intensity.
  - `truecolor: bool` or palette fallback policy if a later terminal compatibility mode is wanted.
  - `missing_texture_policy` such as `Fallback`, `Warn`, or `Error` at load time.
- Refactor `src/render/raster.rs` to support per-cell painting during triangle fill. Preferred shape:
  - Keep simple `fill_triangle(..., ch, style)` for untextured solid rendering.
  - Add a textured/general variant that computes barycentric weights and invokes a small closure or `TrianglePainter` with depth and barycentric coordinates.
- In `src/render/pipeline.rs`, when a face has valid UVs and a texture is selected:
  - During fan triangulation, gather projected vertices plus matching UVs.
  - Interpolate UV per terminal cell using barycentric coordinates.
  - Sample the texture image.
  - Compute glyph brightness from lighting and/or sampled luminance.
  - Write `Style::default().fg(Color::Rgb(r, g, b))`, preserving configured background style.
- For wireframe/points modes, use material/texture average color or current style rather than expensive per-edge sampling unless a simple endpoint interpolation is cheap.
- Ensure large textures do not clone per frame. Store decoded pixels in `Arc<[u8]>` or equivalent and pass references through render code.

Example CLI UX:
- Update `examples/viewer.rs` argument parsing without adding a heavy dependency unless desired. Accept:
  ```bash
  cargo run --release --example viewer --features "cli-example textures" -- \
    models/model.obj --texture models/AXEE_LP_exported_Base_color.jpg
  ```
- Also support MTL-driven textures:
  ```bash
  cargo run --release --example viewer --features "cli-example textures" -- assets/character.obj
  ```
- Show status text indicating texture mode, texture path/count, and fallback reason when no texture is active.
- Keep existing keyboard color toggle cycling through Off/Material/Lighting/Texture/Auto when the `textures` feature is enabled.

Documentation updates:
- README should add a “Textured OBJ” section with:
  - OBJ + MTL + `map_Kd` example.
  - OBJ + manual `--texture` example for missing MTLs.
  - The exact current-model command once implemented:
    ```bash
    cargo run --release --example viewer --features "cli-example textures" -- \
      models/model.obj --texture models/AXEE_LP_exported_Base_color.jpg
    ```
  - Note that the image may be PNG/JPEG and decoding sniffs content, so mislabeled extensions can work.
  - Note that UVs are required for accurate texture placement.
- Update `docs/wiki/Model-Formats.md` and `docs/wiki/Configuration.md` with texture feature flags, supported combinations, and fallbacks.
- Add roadmap notes for future UV generation, normal maps, palette quantization, and benchmark/performance work.

Risks and mitigations:
- Texture sampling can be slower on large meshes: keep feature optional, avoid per-frame texture clones, retain `max_faces`, and add nearest-neighbor as the default fast sampler.
- MTL grammar is broad: parse the common `map_Kd` path robustly and ignore unsupported statements with documented limitations.
- Terminal truecolor support varies: use `Color::Rgb` by default and document terminal requirements; material/lighting modes remain fallbacks.
- Serde derives may not apply cleanly to loaded pixel buffers or `PathBuf`-heavy texture structs: gate/skip serialization for runtime texture pixels or serialize only texture references.

- `cargo fmt --all -- --check`
- `cargo clippy --all-targets --all-features -- -D warnings`
- `cargo test --all-features`
- `cargo check --examples --all-features`

Focused automated tests:
- OBJ parser:
  - Parses `vt` records.
  - Parses face forms `v`, `v/vt`, `v//vn`, `v/vt/vn`.
  - Preserves UV index order through polygon faces.
  - Supports negative vertex/UV/normal indices.
- MTL parser:
  - Parses `Kd` and `map_Kd`.
  - Resolves relative texture paths against the MTL directory.
  - Ignores common `map_Kd` option flags while still extracting the filename.
- Texture loader behind `textures`:
  - Decodes PNG and JPEG into RGBA8.
  - Decodes an image with mismatched extension by content sniffing.
  - Reports clear errors for missing or unsupported images.
- Renderer:
  - Renders a tiny textured triangle into a fixed Ratatui buffer and asserts expected RGB foreground colors at selected cells.
  - Verifies untextured meshes still render under Material/Lighting modes.
  - Verifies missing UVs fall back rather than panic.
- Example/build:
  - `cargo check --example viewer --features "cli-example textures"`.
  - Manual smoke test with `models/model.obj --texture models/AXEE_LP_exported_Base_color.jpg`.

# Delivery Plan
1. Extend mesh and loader contracts for UVs/textures
The crate preserves OBJ UV/normal face data and exposes idiomatic texture load options.
- Update `src/model.rs` with `Vec2`, face UV/normal index storage, and texture/material metadata that does not affect STL or untextured OBJ paths.
- Add `MeshLoadOptions` and keep `Mesh::load` as the compatibility default while adding `Mesh::load_with_options` / texture override APIs behind `textures` where needed.
- Extend `src/error.rs` with texture-specific load/decode variants and clear path context.
- Add parser tests for `vt`, `vn`, `v/vt/vn`, `v/vt`, `v//vn`, negative indices, and missing-UV fallback cases.

2. Implement material texture resolution and image decoding
OBJ + MTL `map_Kd` and OBJ + manual texture override both load decoded RGBA textures.
- Add optional `image` dependency and `textures` feature in `Cargo.toml`; use content-based format guessing so mislabeled files like `AXEE_LP_exported_Base_color.jpg` containing PNG data can decode.
- Add `src/loader/texture.rs` for PNG/JPEG decode to RGBA8 without per-frame cloning.
- Extend `src/loader/mtl.rs` to parse `map_Kd`, preserve `Kd`, and resolve paths relative to MTL files.
- Wire `src/loader/obj.rs` to attach MTL textures when available and attach `MeshLoadOptions::texture_override` as the default texture for UV-only/no-MTL models like `models/model.obj`.

3. Add textured terminal rasterization
The renderer can color terminal cells from interpolated texture samples while retaining current ASCII lighting behavior.
- Extend `src/config.rs` with texture color modes and sampling options such as nearest/bilinear filtering, wrap/clamp policy, V-flip, and lighting multiplication.
- Refactor `src/render/raster.rs` to expose barycentric per-cell painting for textured triangles while keeping existing untextured helpers.
- Update `src/render/pipeline.rs` to select per-face textures, interpolate UVs, sample RGBA pixels, choose glyphs from luminance/lighting, and write `Color::Rgb` styles to the Ratatui buffer.
- Add deterministic renderer tests for a tiny textured triangle, plus regression tests proving material/lighting/STL rendering still works.

4. Upgrade example UX and public docs
Users can run both MTL-textured models and the current no-MTL model through the example with clear documentation.
- Update `examples/viewer.rs` to accept `--texture <path>` and display active texture/fallback status in the help/status panel; color toggles should include texture modes when compiled with `textures`.
- Document release-mode commands, including `models/model.obj --texture models/AXEE_LP_exported_Base_color.jpg`, in `README.md`.
- Update `docs/wiki/Model-Formats.md`, `docs/wiki/Configuration.md`, and `docs/wiki/Getting-Started.md` with supported texture combinations, feature flags, UV requirements, and troubleshooting for missing MTLs.
- Validate with formatting, clippy, tests, docs, example checks, and a manual smoke run against the `models/` asset pair.
