# Requirements

- Support only these mesh inputs: `.gltf`, `.glb`, and `.obj`. Remove STL from the loader path, documentation, error messages, and feature set.
- Keep OBJ companion MTL support because MTL is not a standalone mesh format and is necessary for practical OBJ compatibility. Avoid adding a dependency unless it measurably improves model-file compatibility while reducing or replacing local parser code.
- Prioritize rendering correctness and developer UX over feature bloat.
- Fix Shantae-class glTF issues:
  - hair not rendering / rendering transparent or wrong-colored,
  - eyes flattening to white,
  - eyebrows missing,
  - double-sided/alpha/emissive glTF materials not being represented in the renderer.
- Improve ASCII rendering quality using researched techniques, but keep enhancements opt-in or non-breaking: no forced visual defaults.
- Strengthen tests around glTF/GLB/OBJ compatibility, especially material/texture edge cases, without relying on network downloads in normal CI/local test runs.

# Technical Design

Verified current state:

- `Cargo.toml` currently defaults to `obj`, `stl`, and `mtl`; `gltf` and `textures` are optional. This conflicts with the requested `.gltf`/`.glb`/`.obj`-only support.
- `src/loader/mod.rs::load_with_options` dispatches `.gltf`/`.glb` to `src/loader/gltf.rs::load_gltf`, `.obj` to `src/loader/obj.rs::load_obj_with_options`, and `.stl` to `src/loader/stl.rs::load_stl`.
- `src/error.rs::Error::UnsupportedFormat` still advertises `.stl`.
- `src/loader/gltf.rs` already uses `gltf::import`, which is good: no manual glTF parser bloat. It imports positions, normals, UV set 0, indices, node transforms, animations, skinning, PBR base-color factor, and base-color texture references.
- `src/model.rs::Material` only stores `diffuse` and `diffuse_texture`; it has no alpha mode, alpha cutoff, double-sided, or emissive data.
- `src/render/pipeline.rs::render_mesh` computes projected vertices, uses a z-buffer, triangulates polygons, supports solid/wire/point modes, material/texture/lighting colors, and skips texture samples with `rgba[3] < 16`.
- `src/render/pipeline.rs` applies global `backface_culling` without considering glTF `doubleSided` materials.
- `src/render/color.rs` maps material/texture/lighting color but cannot account for emissive glTF textures/factors.
- `src/render/raster.rs::fill_triangle_with` already has barycentric weights and depth interpolation, which is the right insertion point for better edge/coverage/glyph decisions.

Shantae evidence from `models/shantae/scene.gltf`:

- It has 9 materials and 15 images/textures.
- `MAT_HAIR` has `doubleSided: true` and a base-color texture at texture index 3 / image `textures/MAT_HAIR_baseColor.jpeg`.
- `MAT_EYE_R` and `MAT_EYE_L` have `alphaMode: BLEND`, `doubleSided: true`, base-color texture index 9, and emissive textures.
- Eye whites have a base-color texture and emissive texture.
- The glTF contains named hair, eye, eyelid, and eyebrow nodes, so missing parts are likely material/rendering semantics rather than absent asset data.

Proposed design:

1. Format-scope cleanup
   - Remove `stl` from feature defaults and load dispatch.
   - Remove or isolate `src/loader/stl.rs` and tests from the supported build.
   - Update docs/examples to say OBJ + glTF/GLB only.

2. Material model hardening
   - Extend `Material` with lightweight, dependency-free fields for:
     - `alpha_mode` enum or equivalent (`Opaque`, `Mask`, `Blend`),
     - `alpha_cutoff: f32`,
     - `base_color_alpha: f32`,
     - `double_sided: bool`,
     - `emissive: [f32; 3]`,
     - `emissive_texture: Option<TextureRef>`.
   - Populate these from `gltf::Material` in `collect_materials`.
   - Preserve OBJ/MTL simplicity: OBJ materials default to opaque, one-sided, no emissive.

3. Texture loading and references
   - Continue using the existing `gltf` and optional `image` crates.
   - Make glTF image loading align with glTF material usage and `textures` feature, rather than depending on an option named for material texture loading.
   - Keep missing/failed textures non-fatal unless `strict_textures` is set, but make diagnostics clear.
   - Keep manual MTL unless replacing it with a small compatibility crate demonstrably reduces local parser burden.

4. Rendering fixes
   - Respect per-material `double_sided` when applying `Mesh3dConfig::backface_culling`.
   - Make alpha handling material-aware instead of a single hard-coded threshold.
   - Blend or approximate glTF `BLEND` reasonably for terminal cells; document limitations because true order-independent transparency is beyond this widget’s scope.
   - Include emissive contribution in color choice so eyes and other authored details remain visible.

5. Optional ASCII quality improvements
   - The donut-math reference reinforces the current projection + z-buffer + luminance-ramp design; keep that foundation.
   - The Alex Harri article suggests edge-aware and contrast-enhanced ASCII selection rather than treating characters like pixels. Implement only lightweight optional controls using existing raster data: silhouettes, normal/depth discontinuities, and luminance contrast.
   - Avoid new rendering dependencies and avoid forced default style changes.

# Testing

Validation scenarios an implementation agent can run:

- Format support:
  - `.obj`, `.gltf`, and `.glb` dispatch correctly through `Mesh::load` / `loader::load_with_options` when the matching features are enabled.
  - `.stl` returns `UnsupportedFormat` and docs no longer advertise STL.
- glTF material import:
  - Shantae fixture loads `MAT_HAIR` as double-sided with base-color texture metadata.
  - Shantae eye materials load alpha mode, double-sided flag, base-color texture, and emissive texture/factor metadata.
  - Existing animation/skinning tests for Shantae continue passing.
- Rendering behavior:
  - A double-sided textured triangle still renders when facing away while global backface culling is enabled.
  - Opaque, mask, and blend alpha materials produce distinct expected buffer output.
  - Emissive texture/color contribution changes eye-like material output away from flat white when source data contains detail.
  - Existing texture tests (`renders_textured_triangle_color`, `transparent_texture_samples_do_not_paint`) are updated to match material-aware semantics.
- Developer UX:
  - README and `docs/wiki/Model-Formats.md` show the right feature flags and commands for OBJ and glTF/GLB.
  - `cargo test --features "gltf textures obj mtl"` passes.
  - `cargo test --no-default-features --features "gltf textures"` passes.
  - `cargo test --no-default-features --features obj` passes.
  - `cargo run --release --example viewer --features "cli-example gltf textures" -- models/shantae/scene.gltf` is the manual smoke test for hair/eyes/eyebrows.

# Delivery Plan
1. Narrow supported formats and docs
The crate only accepts OBJ, glTF, and GLB while retaining OBJ material support as a compatibility companion, not a fourth mesh format.
- Update `Cargo.toml` features/defaults to remove `stl`; keep `obj`, `mtl`, `gltf`, and `textures` intentionally feature-gated so users can avoid bloat.
- Remove `src/loader/stl.rs` from the public load path and adjust `src/loader/mod.rs` / `src/error.rs` unsupported-format text to list only `.obj`, `.gltf`, and `.glb`.
- Update `src/lib.rs`, `README.md`, `docs/wiki/Model-Formats.md`, and examples/assets references so the developer UX no longer advertises STL.
- Keep the small manual `src/loader/mtl.rs` parser unless a targeted compatibility dependency clearly replaces more code than it adds; harden MTL only for OBJ diffuse color/texture cases needed by this crate.

2. Harden glTF material import
glTF material metadata carries enough information for Shantae-style hair, eyes, eyebrows, and transparent parts to render predictably.
- Extend `src/model.rs::Material` with glTF-relevant fields: alpha mode/cutoff, alpha factor, double-sided flag, and optional emissive color/texture references while preserving simple OBJ defaults.
- Update `src/loader/gltf.rs::collect_materials` to import `alphaMode`, `alphaCutoff`, `doubleSided`, PBR base-color alpha, base-color texture, and emissive texture/factor; make texture references robust when glTF texture indices and image source indices differ.
- Update `src/loader/gltf.rs::load_textures` to load glTF images whenever textures are requested for glTF, rather than requiring an OBJ/MTL-oriented option name to be set; keep `strict_textures` behavior explicit.
- Add focused tests using `models/shantae/scene.gltf` when present: assert `MAT_HAIR` is double-sided with a loaded base-color texture, eye materials retain alpha/emissive metadata, and eyebrow/eye/hair nodes contribute faces.

3. Make rasterization material-aware
The renderer respects glTF material semantics without forcing new visual defaults on users.
- Update `src/render/pipeline.rs` so `backface_culling` is skipped for `Material::double_sided`, fixing Shantae hair/eye/brow planes that are intentionally double-sided.
- Replace the fixed `rgba[3] < 16` transparent skip with material-aware alpha handling: OPAQUE ignores texture alpha for coverage, MASK uses `alpha_cutoff`, and BLEND uses conservative terminal-friendly coverage/blending behavior documented as approximate.
- Incorporate emissive color/texture into texture/material color selection so eye details are not flattened to white when a glTF supplies emissive eye maps.
- Preserve all existing knobs (`ColorMode`, `texture_lighting`, `texture_filter`, `texture_wrap`, `glyph_ramp`, `backface_culling`) and avoid changing defaults beyond bug fixes necessary for material correctness.

4. Add opt-in ASCII quality improvements
Terminal rendering quality improves through optional, documented algorithms inspired by the referenced ASCII-rendering research.
- Add opt-in config fields in `src/config.rs` for edge/contrast enhancement and glyph selection behavior, defaulting off or current behavior so users are not forced into a style.
- In `src/render/raster.rs` / `src/render/pipeline.rs`, use existing z-buffer and barycentric data to expose edge-aware glyph/intensity decisions: stronger silhouettes/triangle boundaries, normal/depth discontinuity emphasis, and stable luminance mapping based on relative luminance.
- Keep the implementation lightweight: no new runtime dependencies for rendering; use the donut-math takeaways already aligned with the current design (perspective projection, z-buffering, luminance ramp) and the Alex Harri takeaways as optional contrast/shape enhancements.
- Add buffer-level tests for transparent/masked triangles, double-sided backface behavior, emissive color contribution, and edge-enhancement toggles.

5. Build compatibility test matrix
Compatibility confidence increases with asset fixtures and clear commands developers can run locally.
- Add/adjust unit and integration-style tests for `src/loader/gltf.rs`, `src/loader/obj.rs`, `src/loader/mtl.rs`, `src/render/pipeline.rs`, and `src/loader/mod.rs` covering glTF/GLB/OBJ-only support and rejected STL.
- Add a small curated model-fixture test list using existing local assets (`models/axe`, `models/cute`, `models/head`, `models/shantae`) plus generated minimal glTF/GLB fixtures; avoid network downloads in normal tests.
- Optionally provide a documented, ignored compatibility test/script for developers to manually run against a larger downloaded glTF sample corpus when they want broader coverage.
- Validate with `cargo test --features "gltf textures obj mtl"`, `cargo test --no-default-features --features "gltf textures"`, `cargo test --no-default-features --features obj`, and the example viewer against `models/shantae/scene.gltf`.
